02 — Pipeline A

Titanfall 1 → APEX

TF1 ships with a modified Source-style format (MDL/VVD/VTX/VVC). rmdlconv handles the conversion in two passes. Animations need to be decompiled, recompiled, then converted with R5-AnimConv. Textures are delivered via VPK.

01Extract Assets 02Convert Model 03Convert Animations 04Textures & VPK 05GUID Patch 06Pack with RePak
01

EXTRACT ASSETS FROM TF1

All TF1 assets live inside .vpk archives. Use Titanfall VPK Tool (recommended GUI tool — supports Respawn's non-standard VPK format) to browse and extract model files. Open the _dir VPK, navigate to models/weapons/<weapon>/ and extract.

Required Files First person (ptpov_): .mdl, .vtx, .vvc, .vvd — no .phy needed.
World model (w_): .mdl, .vtx, .vvc, .vvd, .phy.

Organise extracted files into separate folders per variant — e.g. ptpov_p2011_auto/ and w_p2011_auto/. Textures can be extracted now or later, order doesn't matter.

VVC filesThe .vvc (vertex colour) file is TF1-specific. TF2 and Apex don't use it — rmdlconv handles it during conversion and it can be discarded after the Apex-format files are produced.
02

CONVERT MODEL (TWO PASSES)

rmdlconv can't go TF1→Apex in one shot — it requires two passes: TF1→TF2 first, then TF2→Apex. This is normal. Just drag the .mdl onto rmdlconv.exe or run it from the command line.

Pass 1 — TF1 → TF2:

Shell
# Run rmdlconv on the TF1 .mdl — outputs a .mdl_new alongside it
rmdlconv.exe ptpov_p2011_auto.mdl

This produces a .mdl_new file. Rename it to .mdl — that's your TF2-format model. Keep the original TF1 .mdl safe in a separate folder (you'll need it for Step 3).

Tip — folder organisationCreate a tf2conv/ subfolder. Copy all files there, delete the .mdl_new from the root, remove the original .mdl from tf2conv/, and rename .mdl_new → .mdl inside tf2conv/. This way you always have the originals to fall back to.

Pass 2 — TF2 → Apex:

Shell
# Create an apexconv/ subfolder, copy tf2conv/ files into it, then run:
rmdlconv.exe ptpov_p2011_auto.mdl

# Outputs: .rmdl + .vg (+ .phy if world model)
# Delete: .mdl, .vvc, .vvd, .vtx — keep only .rmdl and .vg
Output files.rmdl is the Apex model descriptor. .vg is the vertex geometry — RePak picks it up automatically alongside the .rmdl, no need to reference it separately in build.json.
03

CONVERT ANIMATIONS

R5-AnimConv needs a compiled .mdl to work with — not the rmdlconv output, but a fresh recompile of the original TF1 model's skeleton and animations. This requires three sub-steps.

3a — Decompile original TF1 .mdl with Crowbar:

Crowbar
# Open Crowbar 0.74, go to Decompile tab
# Set MDL input to your ORIGINAL TF1 .mdl (not the converted one)
# Output to a subfolder e.g. decompiled/
# Use default settings and hit Decompile
# Produces: .qc, reference .smd, animation .smd files

3b — Recompile with studiomdl:

Shell
# Any Source Engine studiomdl works (Portal 2, HL2, etc.)
studiomdl.exe -game "C:\...\Portal 2\portal2" -nop4 ptpov_p2011_auto.qc

# Output: a freshly compiled .mdl in the game's models/ directory
# Move it to a compiled/ subfolder

3c — Run R5-AnimConv:

Shell
R5-AnimConv.exe "compiled\ptpov_p2011_auto.mdl" -sp animseq/weapons/autopistol_tf1 -rp animrig/weapons/autopistol_tf1

# -sp = output path for .rseq sequence files (arbitrary, pick a sensible convention)
# -rp = output path for the .rrig rig file (arbitrary)
# Outputs: .rrig + one .rseq per animation sequence
# Also prints the full RePak JSON arig entry — copy it for build.json
Paths are arbitraryThe -sp and -rp paths can be anything you want. Convention is animseq/<category>/<name> and animrig/<category>/<name>. For titans: animseq/titans/stryder. For characters: animseq/characters/pilot. It just needs to be consistent with what you put in build.json.
04

TEXTURES & VPK

TF1 textures are .vtf files paired with .vmt material definitions. R5Reloaded still supports VMTs natively, so you can use them directly without converting to Apex's native material format.

Extract the texture files from the TF1 VPK using Titanfall VPK Tool — navigate to materials/models/weapons/<weapon>/ and grab the .vmt and all .vtf files (_col, _ilm, _spc, _nml).

Stage them in the correct path structure and create a manifest, then build with revpk:

Folder structure
vpk/
└── materials/
    └── models/
        └── weapons/
            └── p2011_auto/
                ├── p2011_auto.vmt
                ├── p2011_auto_col.vtf
                ├── p2011_auto_ilm.vtf
                ├── p2011_auto_nml.vtf
                └── p2011_auto_spc.vtf
Manifest — englishclient_watery_common.txt
"BuildManifest"
{
    "materials\models\weapons\p2011_auto\p2011_auto_col.vtf"
    {
        "preloadSize"    "0"
        "loadFlags"      "1049857"
        "textureFlags"   "8"
        "useCompression" "1"
        "deDuplicate"    "1"

    // repeat for _ilm (loadFlags 1049857, textureFlags 8)
    // _spc (loadFlags 1281, textureFlags 8)
    // _nml (loadFlags 1049857, textureFlags 0)
    // .vmt (loadFlags 1281, textureFlags 0)
}

# Outputs: englishclient_watery_common.pak000_dir.vpk + client_watery_common.pak000_000.vpk

05

GUID PATCH (VMT FALLBACK)

After rmdlconv, the .rmdl has texture GUIDs pointing to Apex's internal material asset system. Since we're using VMTs instead, these GUIDs need to be zeroed out — the engine then falls back to resolving textures by material path name.

Each mstudiotexture_t_v54 entry in the RMDL header is 12 bytes: 4 bytes for the name string offset (do not touch), and 8 bytes for the GUID (zero these). The header fields to read are at fixed offsets:

RMDL v54 header offsets
// studiohdr_t_v54 layout (all int32 little-endian)
// 0xD0 = numtextures   (int32)
// 0xD4 = textureindex  (int32, absolute file offset to texture entries)
// Each mstudiotexture_t_v54 entry = 12 bytes:
//   bytes 0-3:  sznameindex  (relative string offset — DO NOT TOUCH)
//   bytes 4-11: GUID         (zero these for VMT fallback)
Python — zero_guids.py
import sys, struct

def zero_rmdl_guids(path):
    with open(path, 'rb') as f:
        data = bytearray(f.read())
    numtextures  = struct.unpack_from('<i', data, 0xD0)[0]
    textureindex = struct.unpack_from('<i', data, 0xD4)[0]
    for i in range(numtextures):
        off = textureindex + i * 12
        data[off+4:off+12] = b'\x00' * 8
    with open(path, 'wb') as f:
        f.write(data)
    print(f"Zeroed {numtextures} GUIDs in {path}")

zero_rmdl_guids(sys.argv[1])
Shell
python zero_guids.py ptpov_p2011_auto.rmdl

03 — Pipeline B

Titanfall 2 → APEX

TF2's MDL (v53) is the closest format to Apex's RMDL (v54). A single rmdlconv pass is all you need for the model. Animations are already in a compatible format — R5-AnimConv handles the conversion directly without the Crowbar decompile/recompile step needed for TF1.

01Extract from RPAK 02Convert Model 03Convert Animations 04Textures & VPK 05Pack + Deploy
01

EXTRACT FROM TF2

TF2 model assets are stored in RPAK files. Use Legion+ or RSX to extract RMDLs and related files from the TF2 RPAK archives. TF2 RPAKs are located at Titanfall2/r2/paks/Win64/.

Unlike TF1, TF2 models are already in RMDL format — so there are no .vvd/.vtx/.vvc files. You just need the .mdl (and .phy for world models).

TipTF2 is the best source for Apex porting. Skeletons, animation conventions, and material paths are closely aligned with Apex — you're upgrading the container format, not rebuilding the content.
02

CONVERT MODEL (SINGLE PASS)

TF2 models only need one rmdlconv pass — TF2 (v53) directly to Apex (v54). No intermediate step required.

Shell
rmdlconv.exe mymodel.mdl
# Outputs: .rmdl + .vg (+ .phy if world model)
# No mdl_new rename needed — output files have different extensions
03

CONVERT ANIMATIONS

TF2 animation data is compatible enough with Apex that you can run R5-AnimConv directly on the TF2 .mdl without the Crowbar decompile/recompile step required for TF1.

Shell
R5-AnimConv.exe "mymodel.mdl" -sp animseq/weapons/myweapon -rp animrig/weapons/myweapon
# Outputs: .rrig + .rseq files
# Console output includes the full arig JSON block for build.json — copy it
04

TEXTURES & VPK

TF2 textures are also VMT/VTF and can be used directly via VPK the same way as TF1. Extract the textures from the TF2 RPAK using Legion+, stage them in the correct folder structure, write a manifest, and build with revpk.

Follow the same VPK build process as the TF1 pipeline — the manifest format and revpk command are identical. See Step 4 of the TF1 pipeline for the full details.

GUID Patch still requiredEven though TF2 is a single pass, the .rmdl output will still have texture GUIDs set. Zero them out the same way as TF1 (see TF1 Step 5) if you're using VMT textures.
05

PACK + DEPLOY

Same as TF1 — assemble your assets folder, generate build.json with the arig entry from R5-AnimConv output and the mdl_ entry for your model, run RePak.

Shell
RePak.exe build.json
# Drop build.rpak + build.starpak + build.opt.starpak into:
# R5SDK/mods/yourmod/paks/Win64/
04 — Pipeline C

SOURCE → APEX

HL2, Portal, CS:S, TF2 (game), L4D and other Valve Source games use standard MDL/VVD/VTX. Decompile with Crowbar, optionally work in Blender, recompile with studiomdl, then follow the same rmdlconv + R5-AnimConv + RePak path as TF1/TF2.

01Decompile MDL 02Blender (optional) 03Recompile + Convert 04Textures & Materials 05Pack + Deploy
01

DECOMPILE WITH CROWBAR

Use Crowbar to decompile the Source MDL to SMD/QC source files. This is the same first step as TF1 animations — use Crowbar's Decompile tab with default settings.

Crowbar
# Crowbar Decompile tab → set MDL input → output to a decompiled/ folder
# Produces: .qc, reference .smd, animation .smd files, .phy.smd
Rig ComplexityCharacter rigs from standard Source (HL2 NPCs, TF2 characters) use ValveBiped which differs from Apex's skeleton. Props and weapons are much simpler — many are effectively static meshes with minimal bones.
02

BLENDER (OPTIONAL)

If the model needs skeleton adjustments, mesh cleanup, or retargeting — import the SMD into Blender via Blender Source Tools, make your edits, then export back to SMD.

For simple props and weapons with no deformable rig, you can skip Blender entirely and go straight to recompile.

TipIf you're porting a weapon that you just want to use with its own animations (not paired with an Apex character), skeleton retargeting isn't needed at all. The rig ports as-is.
03

RECOMPILE + CONVERT

Recompile your SMDs using studiomdl (same as TF1 Step 3b), then run rmdlconv and R5-AnimConv on the output. From this point the pipeline is identical to TF1.

Shell
# 1. Recompile SMDs → .mdl
studiomdl.exe -game "..." -nop4 mymodel.qc

# 2. Convert .mdl → Apex .rmdl (may need two passes depending on source game version)
rmdlconv.exe mymodel.mdl

# 3. Convert animations
R5-AnimConv.exe mymodel.mdl -sp animseq/... -rp animrig/...
04

TEXTURES & MATERIALS

Standard Source VMTs and VTFs work the same way as TF1/TF2 — R5Reloaded supports them natively. Stage them via VPK using the same revpk build process. Make sure the material path in the VMT matches what the RMDL expects.

Don't forget to zero the RMDL texture GUIDs (Step 5 of the TF1 pipeline) for VMT fallback to work.

05

PACK + DEPLOY

Same as TF1 Step 6 — assemble assets folder, write build.json, run RePak, deploy to R5SDK mod directory.

Shell
RePak.exe build.json
# Drop .rpak + .starpak + .opt.starpak into R5SDK/mods/yourmod/paks/Win64/
05 — Pipeline D

Blender → APEX

For models from non-Source games (FBX, OBJ, SEModel etc.) — import into Blender, rig and clean up the mesh, export to SMD, then follow the same recompile + rmdlconv + R5-AnimConv + RePak path. Full material rebuild required.

01Import to Blender 02Rig & Cleanup 03Export SMD 04Compile + Convert 05Materials + Pack
01

IMPORT TO BLENDER

Use whatever importer is appropriate for your source format — Blender natively supports FBX and OBJ. For game-specific formats like SEModel (Call of Duty), use community importers such as io_scene_semodel.

TipCheck the scale on import — many game exporters use centimetre units while Blender defaults to metres. Models may appear enormous or tiny. Set scale to 0.01 on FBX import if needed.
02

RIG & CLEANUP

Clean up the mesh as needed — remove duplicate vertices, fix normals, check for inverted faces. If the model has no skeleton (static prop), you can skip rigging entirely. If it has a skeleton and you want it to work with Apex character models, you'll need to retarget the rig to an Apex-compatible skeleton.

For standalone use (the model uses its own rig, not paired with an Apex character), no retargeting is needed.

03

EXPORT SMD

Install Blender Source Tools to get SMD export support. Export the mesh and skeleton as SMD, then write a basic .qc file for studiomdl.

QC — basic template
$modelname "weapons/mymodel/mymodel.mdl"
$body "Body" "mymodel_reference.smd"
$cdmaterials "models/weapons/mymodel/"
$sequence idle "mymodel_idle.smd" fps 30
04

COMPILE + CONVERT

Compile the SMD with studiomdl, then run rmdlconv and R5-AnimConv — same as TF1 Steps 3b and 3c.

Shell
studiomdl.exe -game "..." -nop4 mymodel.qc
rmdlconv.exe mymodel.mdl
R5-AnimConv.exe mymodel.mdl -sp animseq/... -rp animrig/...
05

MATERIALS + PACK

For non-Source assets you'll need to create VMT files manually pointing to your textures. Convert textures to VTF format using VTFEdit or similar, write VMTs, stage via VPK, then pack with RePak as normal.

Full rebuild requiredUnlike TF1/TF2 where you can reuse the original VMTs, assets from other games have no VMTs at all. You need to create them from scratch — one VMT per material, referencing your converted VTF textures.

07 — Troubleshooting

COMMON ISSUES

Runtime crashes, missing textures, and conversion failures — and how to fix them.

Model loads but textures are missing / appears white or black

The texture GUIDs in the RMDL haven't been zeroed. When using VMT textures, the engine needs GUIDs to be all-zero to trigger the VMT path fallback. Run the zero_guids.py script on your .rmdl before packing with RePak. See TF1 Step 5 for the full details.

rmdlconv outputs a .mdl_new but I'm not sure what to do with it

That's the converted model — rename it to .mdl. Delete the original .mdl from that folder first, then rename .mdl_new → .mdl. It's a good idea to keep your originals in a separate folder so you can always restart from a clean state without re-extracting from the VPK.

R5-AnimConv produces no output / errors immediately

R5-AnimConv needs a recompiled .mdl — not the rmdlconv output and not the raw TF1 .mdl. You must decompile the original TF1 .mdl with Crowbar first, then recompile the SMDs with studiomdl to get a fresh .mdl that R5-AnimConv can process. Also make sure R5-AnimConv is run from the same directory as the .mdl.

Apex crashes when loading the model

Most likely the .rmdl or .vg is missing or corrupt, or the asset paths in build.json don't match the actual file locations in the assets folder. Double check: (1) both .rmdl and .vg are in the same assets subfolder, (2) the _path in the mdl_ entry matches exactly, (3) the $animrigs path matches the arig _path.

Model appears but has no animations

Check that all .rseq files are listed in the $sequences array in your build.json arig entry. R5-AnimConv prints the full list to console when it runs — copy that output directly into your JSON. Also verify the animseq/ folder structure inside assets/ matches the paths in the JSON.

revpk build fails or produces an empty VPK

Check your manifest file (englishclient_watery_common.txt) — every path listed must exist in the staging folder relative to the vpk root. Paths use backslashes in the manifest but the actual folder structure should match. Run revpk from the folder containing the manifest and vpk/ staging directory.

TF1 model has two rmdlconv passes but TF2 only needs one — why?

TF1 uses an older MDL format (v52 with VVD/VTX/VVC) that rmdlconv first converts to TF2's MDL (v53) format, and then a second pass converts v53 → Apex RMDL (v54). TF2 is already at v53 so only the second pass is needed. The two-step process for TF1 is intentional and normal.

World model (.phy) doesn't work correctly in Apex

PHY conversion for TF1 and TF2 world models has known issues in some rmdlconv builds — the PHY offset calculation has historically had bugs. Make sure you're on the latest build of rmdlconv. If physics are still wrong, the model may work fine visually but collisions will be off until a fixed build is available.
