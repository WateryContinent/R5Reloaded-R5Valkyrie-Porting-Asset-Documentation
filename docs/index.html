02 — Pipeline A

TF1 → APEX

Titanfall 1 ships with source-style MDL/VVD/VTX files, these have the same extenstion as regular source but they have been modified by respawn. rmdlconv handles the heavy lifting in getting the models in the correct format and ready to use.

01Extract Assets 02Collect Assets 03Run rmdlconv 04Material Paths 05PHY Physics 06Pack with RePak
01

EXTRACT ASSETS FROM TF1

All TF1 assets live inside .vpk archives, generally within the common vpk. Use ReVPK.exe (located within the bin folder of the game installation), Titanfall VPK Tool or ReVPKEdit to extract your desired assets from the Titanfall install to extract model bundles.

Shell
# Using ReVPK, this dumps the entire vpk.
revpk.exe -unpack englishclient_mp_common.bsp.pak000_dir.vpk ./extracted/
NoteBoth ReVPKEdit and the Titanfall VPK Tool are GUI tool which allows you to extract specific assets from the VPKs.

You need all four sibling files: .mdl (header + bones + surfaces), .vvd (vertex data), .vtx (mesh strips + LODs), and optionally .phy for collision. rmdlconv currently has issues with Titanfall 1 and Titanfall 2 .phy conversion, I have resolved this but it has not released it yet due to other issues.

02

ANALYZE THE r1 FORMAT

TF1 MDLs carry a studiohdr version field — r1 files report version 53, whereas Apex r5 expects 54 with an extended header.

Python — header probe
import struct

with open("model.mdl", "rb") as f:
    magic   = f.read(4)          # b"IDST"
    version = struct.unpack("<I", f.read(4))[0]
    print(f"Magic: {magic}, Version: {version}")
    # r1 TF1 = 53 | r2 TF2 = 54 | r5 Apex = 54 (ext. header)
FieldTF1 r1Apex r5
studiohdr version5354 + ext header
VVD vertex format48-byte tangents32-byte packed
VTX strip flagsValve standardModified hw skin
LOD dist fieldsfloat[8]float[5]
03

RUN rmdlconv

rmdlconv reads r1 MDL/VVD/VTX and emits r5-compatible RMDL. Use the latest build — PHY and VVD size field fixes are critical for crash-free loading.

Shell — basic conversion
# Convert a single TF1 model to Apex r5 RMDL
rmdlconv -i ./extracted/tf1/models/weapons/r101.mdl \
          -o ./output/r5/models/weapons/r101.rmdl \
          --from r1 --to r5

# Batch convert entire directory
rmdlconv --batch ./extracted/tf1/models/ \
          -o     ./output/r5/models/ \
          --from r1 --to r5 --verbose
WarningThe VVD size field bug caused crashes with certain TF1 weapons. Ensure rmdlconv correctly writes vertexDataSize — it must equal numVertices × sizeof(mstudiovertex_t) for the target format, not the source.

For characters, use the --bonemap flag with a JSON remap file to patch skeleton differences between TF1 ValveBiped and Apex r5 bones.

JSON — bone remap
{ "ValveBiped.Bip01_Spine": "spine_0", "ValveBiped.Bip01_Spine1": "spine_1", "ValveBiped.Bip01_L_Clavicle": "clavicle_L", "ValveBiped.Bip01_R_Clavicle": "clavicle_R" }
04

MATERIAL PATH CONVERSION

TF1 MDLs reference materials as materials/models/weapons/... VMT paths. Apex uses an internal RPAK material asset system — all surface references need repathing.

Shell — matpath flag
rmdlconv -i model.mdl --from r1 --to r5 \
          --matmap ./matmap.json

# matmap.json format:
# { "materials/models/weapons/r101": "material/models/weapons/r101/r101" }
NoteApex material assets must exist in your RPAK before the model can reference them. Author stub materials in RePak's JSON definition first, then reference their asset paths in the remap.
05

PHY PHYSICS CONVERSION

TF1 .phy files use the IVP Compact Surface format — same as standard Source Engine. rmdlconv's PHY converter handles the critical size-field fix that caused crashes in older builds.

Shell
# PHY is converted alongside the model automatically
rmdlconv -i model.mdl --phy model.phy --to r5

# The converter fixes:
#  - compactsurface header totalTris field
#  - ledge tree offset recalculation
#  - surface data pointer fixup for r5 memory layout
Common CrashIf Apex crashes on model load with phys_InitModelBounds in the callstack, the PHY header's surfaceSize is wrong. Rerun with --phy-fix-size flag.
06

PACK WITH RePak

With converted RMDLs ready, define a RePak JSON manifest and build the final RPAK for R5Reloaded / R5SDK.

JSON — RePak definition
{ "name": "tf1_weapons_port", "version": 8, "assets": [ { "$type": "rmdl", "path": "models/weapons/r101/r101.rmdl", "settings": { "animrigs": ["models/weapons/r101/r101.rrig"], "sequences": ["models/weapons/r101/r101.rseq"] } }, { "$type": "material", "path": "material/models/weapons/r101/r101_col", "settings": { "surface": "metal" } } ] }
Shell — build RPAK
RePak ./tf1_weapons_port.json
# Output: tf1_weapons_port.rpak + tf1_weapons_port.starpak
# Drop both into: ApexLegends/r5/mods/yourmod/paks/Win64/
03 — Pipeline B

TF2 → APEX

Titanfall 2's r2 RMDL is the closest format relative to Apex's r5. No intermediate FBX step needed for most assets — direct conversion is possible.

01Extract from RPAK 02r2 vs r5 Diffs 03Run rmdlconv 04Animation Sequences 05Pack + Deploy
01

EXTRACT FROM TF2 RPAK

TF2 assets are stored in RPAK files at Titanfall2/r2/paks/Win64/. Use RSX with ZSTD support to extract r2 RMDLs, sequences, and rigs.

Shell
# RSX with ZSTD (compressed RPAKs)
rsx --rpak         Titanfall2/r2/paks/Win64/mp_common.rpak \
    --extract-type rmdl \
    --output       ./tf2_extracted/models/

# Extract all model types including rseq + rrig
rsx --rpak *.rpak --extract-type rmdl,rseq,rrig \
    --output ./tf2_extracted/ --zstd

TF2 RPAKs may be compressed with ZSTD at block level or per-entry. RSX handles both modes automatically when --zstd is passed.

02

r2 vs r5 HEADER DIFFS

TF2 and Apex share RMDL as the base format but diverge in critical header fields. rmdlconv specifically addresses these deltas:

FieldTF2 r2Apex r5 Expected
studiohdr2 flags0x0040 setDifferent flag mask
Max LOD slots75 (truncate excess)
Hardware bone limit53 per draw53 (same)
Flex descriptor formatr2 morphr5 morph (patched)
Extended header size0x0D80x0E4
TipTF2 is the best source for Apex porting. Skeletons, LOD strategies, and material conventions were carried forward almost intact into Apex. You're upgrading the container, not rebuilding the content.
03

RUN rmdlconv (r2 → r5)

Shell
# Single model, r2 → r5
rmdlconv -i ./tf2_extracted/models/weapons/hemlok.rmdl \
          -o ./r5_models/weapons/hemlok.rmdl \
          --from r2 --to r5

# With LOD reduction to 5 max (Apex limit)
rmdlconv -i model.rmdl --from r2 --to r5 --max-lod 5

# Batch directory conversion
rmdlconv --batch ./tf2_extracted/models/ \
          -o     ./r5_models/ \
          --from r2 --to r5 \
          --max-lod 5 --fix-materials
LOD NoteTF2 models with 6–7 LODs need the excess stripped. --max-lod 5 drops the highest-numbered (lowest-detail) LODs. Validate visually — some models have important silhouette data in LOD4.
04

ANIMATION SEQUENCES (RSEQ)

TF2 animation data is in RSEQ files — same format as Apex, with minor version deltas. The most direct asset type to port from TF2.

Shell — RSEQ conversion
# Convert RSEQ alongside the model in one pass
rmdlconv -i hemlok.rmdl \
          --rseq ./tf2_extracted/animations/hemlok_animations.rseq \
          --from r2 --to r5

# RSEQ bone indices are patched automatically when --bonemap is used

TF2 viewmodel animations translate cleanly to Apex weapons when the skeleton is preserved. Third-person character animations may need minor frame timing adjustments for Apex's tick rate.

05

PACK + DEPLOY

TF2 material paths are already close to Apex conventions — typically needing only a prefix repath rather than a full rebuild.

JSON + Shell
// tf2_hemlok_port.json
{
  "name": "tf2_hemlok_port",
  "version": 8,
  "assets": [
    { "$type": "rmdl", "path": "models/weapons/hemlok/hemlok.rmdl" },
    { "$type": "rseq", "path": "animseqs/weapons/hemlok/hemlok_anims.rseq" }
  ]
}

# Build
RePak ./tf2_hemlok_port.json
# Place .rpak in: ApexLegends/r5/mods/yourmod/paks/Win64/
04 — Pipeline C

SOURCE → APEX

HL2, Portal, CS:S, TF2 (game), L4D and other Valve Source games use standard MDL/VVD/VTX. Longest pipeline — requires intermediate FBX export and skeleton retargeting.

01Decompile MDL 02FBX Export 03Skeleton Retarget 04Material Rebuild 05Compile + Pack
01

DECOMPILE WITH CROWBAR

Standard Source MDLs can't be fed directly into rmdlconv. Use Crowbar or mdldecompiler to extract SMD/QC source files first.

Shell — Crowbar CLI
CrowbarCommandLineApp \
    -p ./source_game/models/props/barrel.mdl \
    -o ./decompiled/barrel/
# Produces: barrel.qc, barrel_reference.smd, barrel_phys.smd
Rig ComplexityCharacter rigs from standard Source (HL2 NPCs, TF2 characters) use ValveBiped which differs substantially from Apex's skeleton. Props and weapons are much easier — many have minimal bones or are effectively static meshes.
02

BLENDER FBX EXPORT

Import the SMD into Blender via Blender Source Tools, inspect the mesh and rig, then export as FBX for rmdlconv ingestion.

Python — Blender batch SMD→FBX
import bpy, os

SMD_DIR = "/path/to/decompiled/"
OUT_DIR = "/path/to/fbx_out/"

for f in os.listdir(SMD_DIR):
    if not f.endswith(".smd"): continue
    bpy.ops.wm.read_factory_settings(use_empty=True)
    bpy.ops.import_scene.smd(
        filepath=os.path.join(SMD_DIR, f))
    out = os.path.join(OUT_DIR, f.replace(".smd",".fbx"))
    bpy.ops.export_scene.fbx(
        filepath=out,
        use_mesh_modifiers=True,
        add_leaf_bones=False,
        bake_anim=False
    )
03

SKELETON RETARGETING

For characters, the ValveBiped skeleton must be retargeted to an Apex-compatible rig. rmdlconv accepts FBX with a --skeleton-target reference RMDL for automatic retargeting.

Shell
# Use an existing Apex model as the skeleton reference
rmdlconv --fbx  ./fbx_out/hl2_soldier.fbx \
          --skel ./apex_reference/medium_pilot.rmdl \
          -o    ./r5_models/hl2_soldier.rmdl \
          --to   r5 --retarget

# For props / static meshes with no deformable rig:
rmdlconv --fbx ./fbx_out/barrel.fbx --to r5 --static
TipStatic props with no skeleton are the easiest Source assets to port. --static skips all bone processing — ideal for environment props, crates, and vehicles.
04

MATERIAL REBUILD

Source VMT materials are incompatible with Apex's RPAK material system. New materials must be authored using Apex texture conventions: col (albedo), nml (normal), spc (specular), gls (gloss).

JSON — Apex material def
{ "$type": "material", "path": "material/models/props/barrel/barrel_col", "settings": { "surface": "metal", "shader": "skn_r5", "textures": { "col": "texture/models/props/barrel/barrel_col", "nml": "texture/models/props/barrel/barrel_nml", "spc": "texture/models/props/barrel/barrel_spc" } } }

Convert VTF textures to DXT1/DXT5 DDS, or use RePak's txtr asset type which handles conversion from standard image formats (PNG/TGA) directly.

05

COMPILE TO RMDL + PACK

Shell — full pipeline
# 1. FBX → RMDL (static prop)
rmdlconv --fbx barrel.fbx --to r5 --static \
          -o ./r5_models/models/props/barrel/barrel.rmdl

# 2. Pack everything into RPAK
RePak ./barrel_port.json

# 3. Deploy to R5SDK mod directory
cp barrel_port.rpak "$R5SDK_MOD_DIR/paks/Win64/"
cp barrel_port.starpak "$R5SDK_MOD_DIR/paks/Win64/"

06 — Troubleshooting

COMMON ISSUES

Runtime crashes, visual glitches, and conversion failures — and how to resolve them.

Apex crashes immediately when model is loaded or spawned

Most likely cause: PHY size field mismatch or invalid RMDL extended header. Check the callstack — if it contains phys_InitModelBounds, the PHY is corrupted. Re-run rmdlconv with --phy-fix-size.

If the crash is in CModelLoader::LoadModel, the RMDL header itself is wrong. Verify studiohdr version is 54 and extended header size is 0x0E4.

Model appears inside-out or normals are flipped

TF1/Source MDL winding order is sometimes inverted relative to r5. Pass --flip-normals to rmdlconv, or in Blender select all faces and use Mesh → Normals → Flip before exporting FBX. Common on props originally authored for DX9.

Skeleton deforms incorrectly / mesh explodes during animation

Bone remap is incomplete. Ensure your remap JSON covers all bones in the source skeleton. Unmapped bones default to identity transform, causing vertices to fly to the origin. Use rmdlconv's --dump-bones flag on both source and target to diff the bone lists.

Materials appear black or model has no textures

The material asset path referenced in the RMDL doesn't exist in the loaded RPAK. Verify the path string in your converted RMDL exactly matches the "path" field in your RePak JSON — these are case-sensitive on Linux. Also check that the .starpak streaming file is present alongside the .rpak.

RSX fails to extract with "decompression error"

The RPAK uses ZSTD compression. Ensure RSX was built with -DUSE_ZSTD=ON in CMake, and pass --zstd explicitly if auto-detection isn't working. Some TF2 RPAKs use per-entry ZSTD — older RSX builds only handled block-level compression.

rmdlconv outputs "LOD count exceeds r5 maximum"

TF1/TF2 models can have up to 7–8 LOD levels. Apex's r5 RMDL supports max 5. Add --max-lod 5 to your command. The converter drops the highest-numbered (lowest-poly) LOD slots. Use --lod-select 0,1,2,3,4 for manual control over which indices to keep.

VVD size field warning during conversion

A known bug in older rmdlconv builds where vertexDataSize was calculated using source-format vertex size instead of target-format. Update to the latest build. The field must equal numVertices × sizeof(mstudiovertex_r5_t).

RePak builds successfully but model doesn't appear in-game

Check R5SDK's mod loading output to confirm the RPAK is actually being loaded. If it is, the asset path referenced in your Squirrel script doesn't match what's in the RPAK — paths are case-sensitive and must not have a leading slash. Use R5SDK's asset browser to confirm registered paths after RPAK load.
